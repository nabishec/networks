# ARQ — Automatic Repeat reQuest

## TL;DR
Общий принцип: получатель **подтверждает** (ACK) полученный фрейм; если отправитель не получает ACK за разумное время — **повторяет** передачу. Из этой идеи строятся все надёжные протоколы: [[Stop-and-Wait]], [[Go-Back-N]], [[Selective Repeat]], а на L4 — TCP. ARQ ≠ FEC: первый просит повторить, второй — исправляет ошибки на месте.

## Какую проблему решает
Канал теряет фреймы (помеха, переполнение буфера маршрутизатора, разрыв связи). Без какой-либо обратной связи отправитель не знает, что не дошло. ARQ — простая идея: получатель **сообщает**, что получил, и если отправитель долго ничего не слышит — шлёт ещё раз. Это даёт **надёжную доставку** поверх ненадёжного канала.

## Как работает

**Минимальные ингредиенты:**
1. **Нумерация фреймов (seq):** чтобы различать новые от дубликатов.
2. **Подтверждения (ACK):** получатель сигнализирует, что получил.
3. **Таймер:** отправитель ждёт ACK; не дождался → шлёт повторно.
4. **Буфер:** отправитель хранит неподтверждённые фреймы; получатель — фреймы, пришедшие не по порядку (если поддерживает).

**Варианты протоколов отличаются по:**
- Размеру окна отправителя (Stop-and-Wait = 1; sliding window > 1).
- Размеру окна получателя (Go-Back-N = 1; Selective Repeat > 1).
- Стратегии повтора (cumulative vs selective ACK).

**ARQ vs FEC:**

| Свойство | ARQ | FEC (Forward Error Correction) |
|---|---|---|
| Что делает | просит повторить | исправляет на месте |
| Требует обратного канала | **да** | нет |
| Overhead | низкий, только если ошибки | постоянный (избыточные биты) |
| Подходит для | пакетных каналов с ACK | вещательных, real-time, спутник в одну сторону |
| Примеры | TCP, HDLC, 802.11 ACK | [[Корректирующие коды — Хэмминг]], [[Код Рида-Соломона]], [[LDPC и сверточные коды]] |

**Гибрид (HARQ):** в LTE/5G — Hybrid ARQ. FEC + ARQ: каждый фрейм имеет встроенный код-корректор + повтор при сильной потере. Сочетает лучшее.

## Пример
- **TCP:** ARQ на L4. Каждый сегмент нумеруется (seq), получатель шлёт ACK (с seq последнего непрерывно полученного). Таймер на стороне отправителя; при timeout — retransmit.
- **Wi-Fi (802.11):** ARQ на L2. Каждый data-фрейм требует ACK от приёмника; не пришёл за SIFS → передатчик повторяет. Без этого — Wi-Fi был бы катастрофически непригоден из-за частых потерь в эфире.
- **HDLC, PPP:** ARQ через I-frame с seq и обратным ACK (numbered mode).

## Связи
- **Базируется на:** [[Канальный уровень]] (классический хост ARQ), [[Надёжность службы]] (это её механизм).
- **Используется в:** [[Stop-and-Wait]], [[Go-Back-N]], [[Selective Repeat]] — конкретные протоколы; [[TCP]] на L4.
- **Соседи по уровню:** **FEC** ([[Корректирующие коды — Хэмминг]] и др.) — альтернатива/дополнение; **HARQ** в LTE/5G — гибрид.
- **Противопоставляется:** **best-effort** — без ARQ, без гарантий (UDP, Ethernet без L2-ACK).

## Подводные камни
- ARQ повышает **средний throughput**, но не помогает при **сильных** потерях (>10%) — там лучше FEC или гибрид.
- **End-to-end vs hop-by-hop ARQ:** TCP делает ARQ end-to-end. L2-ARQ (Wi-Fi) — только на одном hop. Принцип end-to-end argument: для надёжности приложению нужен e2e-ARQ. Hop-by-hop — оптимизация.
- ARQ требует **обратного** канала. На спутниковых вещательных каналах (одна антенна → миллионы) обратного канала нет — там используют FEC или гибрид.
