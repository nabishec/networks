# DNS — кэширование и TTL

## TL;DR
Каждая DNS-запись несёт **TTL** (Time To Live, секунды). Recursive resolver и stub-resolver **кэшируют** ответ на TTL-секунд. Без кэша корневые и TLD-серверы умерли бы под нагрузкой. С кэшем 99% запросов — мгновенные. Цена — **stale data** при изменении IP до истечения TTL.

## Какую проблему решает
DNS-запросов в интернете — миллиарды в секунду. Без кэша вся нагрузка падала бы на authoritative-серверы. С кэшем большинство запросов резолвятся локально в долях миллисекунды.

## Как работает

**Уровни кэша:**

| Уровень | TTL-наблюдение | Где |
|---|---|---|
| **Browser** | сотни секунд | Chrome/Firefox per-tab |
| **OS stub** | DNSCache (Windows), nscd (Linux), mDNSResponder (macOS) |
| **Recursive resolver** | главный keeper | 8.8.8.8, провайдер |
| **Authoritative** | не кэширует, source of truth |

**TTL значения:**
- **300** (5 мин) — динамический контент, частые изменения.
- **3600** (1 ч) — стандарт для большинства A-записей.
- **86400** (24 ч) — стабильные NS-записи.
- **604800** (7 дней) — у root.

**Стратегии:**
- **Перед миграцией:** уменьшаем TTL до 60 секунд за неделю до изменения; меняем IP; через час возвращаем большой TTL.
- **Negative caching** (RFC 2308): если запись «не существует» (NXDOMAIN) — кэшируется тоже, обычно SOA negative TTL (5 мин).

## Пример
**Сайт меняет хостинг:**
- Текущая A `93.184.216.34` (старый), TTL=86400 (24ч).
- За неделю до миграции: уменьшить TTL до 300.
- Через 24 часа после уменьшения **большинство** мира будет иметь TTL=300 в кэше.
- В день миграции: переключить A на новый IP. Через 5 минут все увидят новый.
- Через неделю: вернуть TTL=86400 для load reduction.

**Negative cache:** опечатка в URL `wikipedia.ogr` — DNS отвечает NXDOMAIN, кэшируется на 5 мин → следующий ту же опечатку быстро.

## Связи
- **Базируется на:** [[DNS]] (механизм), TTL-поле в каждой RR ([[DNS — типы записей]]).
- **Используется в:** все DNS-разрешения; [[CDN — устройство]] (короткие TTL для request-routing); [[Anycast]] (комбинированный).
- **Соседи по уровню:** **negative caching** (NXDOMAIN), **DNS pre-fetch** в браузерах.
- **Противопоставляется:** «без кэша» — нереалистично.

## Подводные камни
- **TTL — рекомендация, не жёсткое правило.** Некоторые resolver'ы держат запись дольше; некоторые истекают раньше. Нет гарантий.
- **Browser DNS cache** часто **жёстче, чем TTL** — Chrome держит 60 секунд минимум независимо от TTL.
- **DNS pre-fetch** в браузерах: для ссылок на странице делает резолв заранее → ускоряет navigation, но утечка для аналитики.
- **Stale DNS** — главная причина «мне сайт не работает, а другим работает» после миграции.
