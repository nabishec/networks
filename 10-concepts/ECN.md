# ECN — Explicit Congestion Notification (RFC 3168)

## TL;DR
Расширение IP+TCP, позволяющее маршрутизаторам **маркировать** пакеты вместо их **сброса** при перегрузке. Получатель отражает сигнал отправителю, который **замедляется** (как при потере), но **без потери пакета**. Радикально снижает задержки и потери в перегруженных сетях, особенно в DC.

## Какую проблему решает
Классический TCP узнаёт о перегрузке через **потерю** пакета (drop tail или RED). Но потеря — это:
- Потеря данных → ретрансмиссия → задержка.
- Не различает «настоящая ошибка канала» и «перегрузка»: TCP реагирует одинаково.

ECN отделяет: **«сеть перегружена»** маркируется в самом пакете, доставленном к получателю. TCP замедляется без переотправок.

## Как работает

**В IP-заголовке** — 2 бита ECN (часть TOS/DSCP):
- `00` — Not-ECN-capable.
- `01` (ECT(1)), `10` (ECT(0)) — Capable, маршрутизатор может маркировать.
- `11` (CE — Congestion Experienced) — маршрутизатор маркировал.

**В TCP-заголовке** — 2 флага:
- **ECE (ECN-Echo)** — получатель сообщает «я видел CE-пакет, ты, отправитель, медленнее».
- **CWR (Congestion Window Reduced)** — отправитель подтверждает «я уменьшил cwnd».

**Цикл:**
1. Отправитель шлёт пакет с ECT=10.
2. Маршрутизатор видит перегрузку (RED-style) → маркирует CE (вместо drop).
3. Получатель видит CE → шлёт следующий ACK с флагом ECE.
4. Отправитель видит ECE → уменьшает cwnd (как при потере) → шлёт следующий пакет с CWR.
5. Получатель видит CWR → перестаёт ECE'кать (одной реакции достаточно).

## Пример
**DC-сеть с DCTCP:**
- Пороги ECN на switch'ах настроены агрессивно (low queue threshold).
- Сервера маркируют ECN-capable.
- При лёгкой перегрузке switch маркирует, не дропает.
- DCTCP считает **долю** маркированных пакетов и пропорционально снижает cwnd.
- Результат: микросекундные задержки, ноль потерь, высокий throughput. Невозможно с обычным TCP+drop tail.

**В публичном интернете:** ECN недо-развёрнут. Многие старые firewall'ы блокируют ECN-флаги. **L4S** (Low Latency Low Loss Scalable, RFC 9330+) — современная попытка снова распространить ECN-подобную сигнализацию end-to-end.

## Связи
- **Базируется на:** [[Перегрузка сети]], [[RED и AQM]] (механизм маркировки), [[IPv4]] / [[IPv6]] (поля ECN), [[TCP]] (флаги ECE/CWR).
- **Используется в:** DCTCP, L4S, BBRv3, modern Linux TCP.
- **Соседи по уровню:** drop-based congestion signal — традиционный путь.
- **Противопоставляется:** **drop как сигнал** — потеря данных + задержка; ECN — без потери.

## Подводные камни
- ECN требует поддержки **обеих** сторон + промежуточных маршрутизаторов. Если хоть кто-то не понимает — fallback к drop.
- Старые middleboxes часто **сбрасывают** ECN-флаги или роняют ECN-помеченные пакеты — bug-friendly path. Из-за этого Linux долго ECN отключал по умолчанию для исходящих.
- В DC ECN включают агрессивно; в публичном интернете осторожнее.
