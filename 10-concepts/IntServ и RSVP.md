# IntServ и RSVP — Integrated Services (RFC 1633)

## TL;DR
Архитектура **per-flow** QoS-резервирования: для каждого потока (например, VoIP-сессии) приложение через **RSVP** (Resource Reservation Protocol) запрашивает **гарантию** ресурсов на пути от источника до получателя. Маршрутизаторы держат **state per-flow**. Не масштабируется на интернет (миллионы потоков → миллионы записей в каждом маршрутизаторе) — почти не используется в публичной сети, но идея жива в [[MPLS]]-TE.

## Какую проблему решает
Best-effort IP не даёт **гарантий**. Real-time приложения (VoIP, видеоконференции) требуют предсказуемой задержки/jitter. IntServ предложил: **резервируем** ресурсы на пути заранее, как телефонный канал в circuit-switching, но поверх IP.

## Как работает

**Классы услуг:**
- **Guaranteed Service** — жёсткие пределы задержки и пропускной способности.
- **Controlled Load** — поведение «как в ненагруженной сети».
- **Best Effort** — обычный.

**Resource ReSerVation Protocol (RSVP, RFC 2205):**
1. **Sender → Receiver:** **PATH** message с описанием потока (Tspec — token bucket parameters).
2. PATH запоминается в каждом маршрутизаторе на пути.
3. **Receiver → Sender:** **RESV** message с запросом ресурсов (Rspec).
4. Каждый маршрутизатор на обратном пути проверяет: есть ли свободная ёмкость? Если да — резервирует.
5. RESV дошёл до отправителя → сессия с гарантиями.
6. **Soft state:** периодические refresh, иначе резерв истекает.

## Пример
- **VoIP-вызов с гарантированным QoS:** клиент шлёт RSVP-резерв на 64 кбит/с, задержку ≤ 100 мс. Каждый маршрутизатор резервирует место в очереди под этот flow.
- На практике в **публичном** интернете это не работает: каждый ISP должен поддерживать RSVP, миллионы flows = катастрофа state.
- **MPLS-TE** (Traffic Engineering) использует RSVP-TE для резервирования **trunk LSP** между маршрутизаторами провайдера — это работает, потому что **trunk'ов** немного, а не миллионы потоков.

## Связи
- **Базируется на:** [[QoS vs QoE]] (это её достижение через резервирование).
- **Используется в:** **MPLS-TE** (RSVP-TE), редко — в корп. сетях для real-time. В публичном интернете отсутствует.
- **Соседи по уровню:** [[DiffServ]] — альтернатива без per-flow state.
- **Противопоставляется:** именно [[DiffServ]] — IntServ per-flow, DiffServ per-class.

## Подводные камни
- **Не масштабируется** на публичный интернет — миллионы потоков, миллионы state'ов.
- Требует поддержки от **всех** маршрутизаторов на пути; единственный без RSVP — резервация ломается.
- Fee/billing для резервирования отсутствуют в стандарте → нет коммерческого стимула для развёртывания.
