# Фрагментация и PMTUD

## TL;DR
Разные L2-сети имеют разные **MTU** (Maximum Transmission Unit): Ethernet 1500, Wi-Fi 2304, PPPoE 1492. Если IP-пакет больше MTU следующего hop'а — нужно его **разбить на фрагменты**. В IPv4 это делает **маршрутизатор** на пути; в IPv6 — **только источник**, через **Path MTU Discovery**. Полная сборка — на **получателе**.

## Какую проблему решает
Интернет состоит из разнородных сетей. Если потребовать «весь интернет = один MTU» — все ограничены минимумом. Если позволить переменный MTU — нужен механизм адаптации. Фрагментация решает: «отправляй любой размер, разрежем при необходимости».

## Как работает

### IPv4 fragmentation
- Маршрутизатор видит пакет N байт, исходящий MTU = M < N.
- Разбивает на фрагменты ≤ M − header.
- Каждый фрагмент:
  - Identification (одинаковый у всех фрагментов одного пакета).
  - **Fragment Offset** (где в исходном пакете был этот кусок).
  - **MF** (More Fragments) — флаг: 1 для всех кроме последнего.
  - **DF** (Don't Fragment) — флаг: запрет фрагментации; нарушение → ICMP-ошибка.
- **Сборка** — только на получателе (промежуточные маршрутизаторы не собирают). Если хоть один фрагмент потерян — весь пакет теряется (нет частичной retx).

### IPv6 — без фрагментации в маршрутизаторах
- Маршрутизатор слишком большой пакет → **дроп + ICMPv6 «Packet Too Big»**.
- Источник узнаёт MTU пути и фрагментирует **сам** (через Fragment Extension Header).
- Это упрощает железо маршрутизаторов.

### Path MTU Discovery (PMTUD)
- Источник шлёт пакет с **DF=1** и максимальным предполагаемым MTU.
- Маршрутизатор не может пропустить → ICMP «Fragmentation needed, MTU=X».
- Источник уменьшает свой MSS/MTU до X и повторяет.
- Конвергируется до min(MTU on path).

## Пример
**Скачивание через VPN:**
- VPN добавляет 20–60 байт overhead → effective MTU < 1500.
- Без PMTUD: пакеты с DF могут не пройти, мозговая боль для пользователя.
- С работающим PMTUD: TCP MSS уменьшается, всё работает.

**Проблема:** многие провайдеры **блокируют ICMP** (по ошибке или ради «безопасности») → PMTUD ломается → "PMTUD black hole". Решение — **MSS clamping** на VPN-граница.

## Связи
- **Базируется на:** [[Сетевой уровень]], [[IPv4]] / [[IPv6]] (где живут флаги), [[ICMP]] (для PMTUD).
- **Используется в:** все межсетевые передачи; VPN, туннели.
- **Соседи по уровню:** [[Туннелирование]] — типичная причина MTU-проблем.
- **Противопоставляется:** «один MTU на весь интернет» — никогда не было реальностью.

## Подводные камни
- **IPv4 fragmentation на маршрутизаторе считается плохой практикой:** дополнительная нагрузка, потеря одного фрагмента = потеря всего, NAT/firewall'ы плохо обрабатывают фрагменты.
- **DF=1 + блокированный ICMP = black hole:** пакет в одну сторону теряется, но отправитель не знает. Симптом — TCP «зависает» после handshake'а.
- **Jumbo frames (9000 байт)** в DC ставят свои сложности на стыке с обычным MTU 1500.
