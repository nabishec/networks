# Piggybacking

## TL;DR
Оптимизация: вместо отправки **отдельного ACK-фрейма**, ACK **встраивается в заголовок** обычного data-фрейма, идущего в обратном направлении. Экономит overhead'ы (заголовок, контрольная сумма, межфреймовые промежутки). Работает только когда есть **двусторонний** трафик; иначе ACK всё равно идёт отдельно.

## Какую проблему решает
ACK-фрейм без полезной нагрузки — это всё равно фрейм с заголовком, footer'ом, межфреймовыми зазорами. На высоких скоростях это десятки байт чистого overhead на каждый ACK. Если в обратном направлении уже идёт data-фрейм, **разумно** засунуть ACK в его заголовок и не плодить отдельный.

## Как работает
**Без piggybacking:**
```
A → B: data(seq=N)
B → A: ACK(N)         ← пустой фрейм с одним полем ACK
A → B: data(seq=N+1)
```

**С piggybacking:**
```
A → B: data(seq=N, ack=...)
B → A: data(seq=M, ack=N)      ← ACK встроен в обратный data-фрейм
A → B: data(seq=N+1, ack=M)
```

Во многих L2/L4-протоколах заголовок имеет два поля: **seq** (что я шлю) и **ack** (что я последним принял). Piggybacking использует это поле в обратном направлении.

**Вопрос таймера:** если в обратном направлении нет data-фрейма за разумное время, нельзя ждать вечно — отправляем «голый» ACK. Так делает TCP: **delayed ACK** — ждёт ~200 мс в надежде на piggyback, иначе шлёт сам.

## Пример
- **TCP:** в каждом сегменте есть и поле seq, и поле ack-num. При двустороннем диалоге (HTTP-запрос ↔ ответ; SSH-сессия) — почти каждый сегмент несёт ACK на пакет в обратную сторону. Голые ACK — только когда долгое одностороннее молчание.
- **HDLC, PPP:** аналогично, ACK-поле встроено в data-фрейм (S-frame в HDLC отдельно, но в I-frame есть N(R) — receive sequence).

## Связи
- **Базируется на:** [[ARQ]], [[Скользящее окно]] — без них нет ACK'ов вообще.
- **Используется в:** [[TCP]] (стандартный приём), [[Go-Back-N]], [[Selective Repeat]] (теоретическая база), HDLC, многие L2-протоколы.
- **Соседи по уровню:** **delayed ACK** (TCP) — родственный приём, дающий время на piggyback.
- **Противопоставляется:** «всегда отправлять ACK сразу» — простая, но расточительная стратегия.

## Подводные камни
- Если обратный data-трафик слабый или отсутствует, piggybacking вырождается в обычные ACK-фреймы. Это **оптимизация common case**, не всегда работающая.
- **Delayed ACK + Nagle** в TCP в плохой комбинации даёт **artificial delay** до 200 мс на маленьких объёмах. *(Алгоритм Нэгла — приём TCP, копящий маленькие записи в один сегмент, чтобы не плодить мини-пакеты.)* Известная проблема: интерактивные приложения (Telnet, REST API) могут заметно «задумываться». Решение: TCP_NODELAY (отключить Nagle) или TCP_QUICKACK.
- В одностороннем bulk-transfer (например, скачивание файла) piggybacking почти не работает — обратно идут только ACK'и без данных.
