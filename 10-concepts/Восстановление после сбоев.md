# Восстановление после сбоев (crash recovery)

## TL;DR
Что делать, если **сервер перезагрузился посередине разговора**? У клиента есть state соединения, у сервера — нет. **End-to-end argument** (Saltzer 1984): уровень N+1 **обязан помнить** state — N не может его восстановить. Tanenbaum в §6.2.6 показывает: при любых стратегиях клиент/сервер существует комбинация, дающая **DUP** (дубликат) или **LOST** (потерю). Идемпотентность операций — единственная надёжная защита.

## Какую проблему решает
Идеальная сеть: сервер никогда не падает. Реальность: rebooten, kernel panic, OOM-killer. Клиент не знает, что **именно** сделал сервер до перезагрузки — выполнил операцию или нет. Запрос повторить → возможно дубликат (deduct money дважды). Не повторять → возможно потеря.

## Как работает

**Допущения:** сервер выполняет «write to file» — атомарную операцию. Сообщает клиенту через ACK. Клиент шлёт следующее.

**Сервер падает между:**
- A — выполнить write.
- W — отправить ACK.

**4 события, 4 порядка:**
- AC, ACK, W (норма).
- A, crash, W never sent.
- A, W, crash после.
- crash до A.

Клиент видит только **«ACK не пришёл»**. Стратегии:
1. **Always retransmit** (AR) — повторить.
2. **Never retransmit** (NR) — забить.
3. Retransmit only если **ACKed** ранее (AS).
4. Retransmit только если **NotAckedYet** (ANA).

**Tanenbaum (стр. 597) показывает таблицу 8 комбинаций:**

| Strategy сервера / Стратегия клиента | AR | NR |
|---|---|---|
| AC(W) — ACK before write | ❌ DUP | ❌ LOST |
| AWC — ACK after write | ✓ OK | ❌ LOST |
| C(AW) — ACK and write atomically | ❌ DUP | ❌ LOST |

В каждой строке хотя бы одна стратегия даёт DUP или LOST.

**End-to-end argument (Saltzer, Reed, Clark 1984):**
> Даже если транспортный уровень делает retransmit, **только** конечное приложение знает, как реагировать на дубликаты. **Реальная** надёжность — на уровне приложения, не на TCP.

## Пример
**Платежная транзакция через REST API:**
- POST /pay {amount: 100} → клиент → сервер.
- Сервер списывает $100 → **crash перед ACK**.
- Клиент: timeout → retry POST.
- Без идемпотентности: списать ещё $100 → проблема.

**Решение — Idempotency-Key** (header в Stripe, AWS):
- Клиент шлёт уникальный UUID в каждом POST.
- Сервер: «уже видел этот UUID — не делать повторно, вернуть тот же ответ».
- AT-LEAST-ONCE + идемпотентность = фактически exactly-once.

## Связи
- **Базируется на:** [[TCP]] (где crash recovery относится), [[Транспортный уровень]].
- **Используется в:** [[RPC]] (idempotency, retry policies), [[HTTP]] (POST/PUT vs GET идемпотентность), distributed systems вообще.
- **Соседи по уровню:** **soft state** (в RSVP, network) — родственная идея «истекает по timer'у».
- **Противопоставляется:** «exactly-once в распределённой системе» — невозможно без 2-phase commit или idempotency.

## Подводные камни
- **Транспортные дубликаты** (TCP) — дешёво поймать через seq numbers.
- **Application-level дубликаты** (после crash) — поймать **намного** сложнее, нужен app-level state (Idempotency-Key, transaction log).
- **Two-army problem:** при ненадёжной сети **нельзя гарантировать** одновременное соглашение двух сторон. Crash recovery — частный случай этой фундаментальной невозможности.
