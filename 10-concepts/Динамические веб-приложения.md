# Динамические веб-приложения

## TL;DR
От статических HTML 1990-х → к **server-side dynamic** (PHP/Python/Java рендерят HTML per-request) → к **AJAX** (XMLHttpRequest, async-запросы из JS, partial обновления страницы) → к **SPA** (Single Page Application — react/vue/svelte; вся логика в JS, общается с backend через API). Современный «веб-сайт» = **frontend SPA** + **backend REST/GraphQL/gRPC API**.

## Какую проблему решает
Чисто статический HTML не подходит для интерактива (банк, соцсеть, корзина). Нужна **обновляемая** под пользователя страница. Эволюция шла от «полная перезагрузка на каждое действие» к «обновление кусками без reload».

## Как работает

**Эпоха server-side rendering (SSR):**
- Каждый клик/форма → POST → сервер генерирует full HTML → отдаёт.
- Полная перезагрузка страницы.
- Технологии: PHP, Java Servlets, Ruby on Rails, Django.

**AJAX (2005, Gmail):**
- JS отправляет async-запрос (`fetch()` или старый XMLHttpRequest).
- Сервер отдаёт JSON (или XML — отсюда «X»).
- JS обновляет часть страницы.
- Без reload → ощущение **приложения**, не сайта.

**SPA (single-page application):**
- Один initial HTML с большим JS-bundle.
- Routing внутри браузера (URL меняется через History API без reload).
- Каждое состояние — данные через API + render через React/Vue/Angular.
- Backend = только API (REST, GraphQL).
- Hyper-фронт + умный API = архитектура 2010-х+.

**Современные подходы:**
- **SSR + hydration:** сервер рендерит initial HTML (быстрый first paint), JS «гидратирует» в SPA. Next.js, SvelteKit, Nuxt.
- **Static Site Generation (SSG):** генерация во время build → CDN. Подходит для контента (блог, документация).
- **Edge functions:** server-side код прямо в CDN (Cloudflare Workers, Vercel Edge).

## Пример
**Twitter/X:**
- Initial load: HTML+JS bundle ~1 МБ.
- Затем: каждое действие (лайк, scroll) → API-запрос → JSON → JS обновляет UI.
- WebSocket для real-time updates.

**Banking:**
- Чаще SSR + classic forms (security, compatibility).
- Приложение менее «отзывчивое», но проще validate.

## Связи
- **Базируется на:** [[Web — архитектура]], [[HTTP]] (или WebSocket/SSE).
- **Используется в:** все современные web-сервисы.
- **Соседи по уровню:** mobile apps — другая платформа, тот же backend API.
- **Противопоставляется:** static HTML — для интерактива не подходит.

## Подводные камни
- **SEO:** SPA hard for crawlers, не видят контент после JS. Решение — SSR.
- **Bundle size:** большой initial JS = долгий first paint. Code splitting, lazy loading.
- **State management** — Redux, MobX, Zustand: сложность фронта быстро растёт.
- **API versioning:** добавил поле — старые клиенты в кэше браузера не понимают. GraphQL смягчает (запрашиваешь явно).
