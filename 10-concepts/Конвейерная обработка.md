# Конвейерная обработка (pipelining)

## TL;DR
Идея: вместо «отправил один фрейм → жду ACK → отправил следующий» (Stop-and-Wait) **отправлять несколько фреймов подряд** без ожидания. Получатель отвечает потоком ACK'ов. Утилизация канала повышается с долей процента до близкой к 100%. Базис всех современных надёжных протоколов (TCP, sliding window).

## Какую проблему решает
[[Stop-and-Wait]] отправитель **простаивает** RTT после каждого фрейма → утилизация на high-BDP сетях ничтожна. Pipelining: пока ждёшь ACK на N=1, **отправь N=2, 3, 4, ...** — и все они в полёте. Пропускная способность × W (размер окна).

В Tanenbaum'е §3.4.1 — отдельная подсекция «Цель: двунаправленная передача, отправка сразу нескольких фреймов».

## Как работает

**Pipelining + ARQ → две стратегии при потере:**
1. **Go-Back-N** ([[Go-Back-N]]) — простая: при потере N, повторить N **и все после**. Окно получателя = 1.
2. **Selective Repeat** ([[Selective Repeat]]) — умнее: повторить **только потерянный** N. Окно получателя > 1.

**Размер окна (W):**
- Идеал: W ≥ [[Bandwidth-Delay Product|BDP]] / frame_size.
- Меньше → недоутилизация.
- Больше → ничего не даёт (всё равно ждёшь ACK).

**Эффективность** vs Stop-and-Wait:
- Stop-and-Wait утилизация ≈ T_frame / (T_frame + RTT). При больших RTT → 0.
- Pipelining utilization = min(1, W·T_frame / (T_frame + RTT)). При W ≥ BDP/frame → ~1.

## Пример
**Канал 100 Мбит/с, RTT 100 мс, фрейм 1500 байт = 12000 бит:**
- T_frame = 0.12 мс.
- Stop-and-Wait: 0.12 / (0.12 + 100) ≈ 0.12% утилизация.
- Pipelining с W=1000: 1000·0.12 / (0.12+100) = 119% (cap'ed at 100%).
- W ≈ 833 (BDP/frame) — точно sufficient.

## Связи
- **Базируется на:** [[Stop-and-Wait]] (предшественник), [[ARQ]].
- **Используется в:** [[Скользящее окно]] (механизм реализации), [[Go-Back-N]], [[Selective Repeat]], [[TCP]].
- **Соседи по уровню:** [[Bandwidth-Delay Product]] (определяет нужный W).
- **Противопоставляется:** Stop-and-Wait — не масштабируется на BDP.

## Подводные камни
- **Sequence numbers wraparound:** при больших W cyc nm должны быть достаточно длинными (W ≤ N/2 для SR, W ≤ N-1 для GBN).
- **Reordering:** в IP-сетях фреймы могут прийти не по порядку → SR должна буферизовать.
- **Burst of ACKs** перегружает обратный канал. Решения: cumulative ACK, delayed ACK.
