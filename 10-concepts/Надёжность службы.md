# Надёжность службы

## TL;DR
Служба **надёжна**, если она гарантирует, что отправленные данные будут доставлены: без потерь, в нужном порядке, без дублей. Иначе — **ненадёжна** (best-effort). Эти гарантии стоят задержки и накладных расходов: для real-time медиа надёжность часто **вредна**.

## Какую проблему решает
Не всем приложениям нужна одинаковая надёжность. Файл — нужен в точности; голос в звонке — лучше пропустить пакет, чем тормозить. Надёжность — это **выбор**, и его делают на разных уровнях стека независимо.

## Как работает
Надёжная служба обычно опирается на:
- **Подтверждения (ACK):** получатель отвечает «дошло».
- **Повторные передачи (ARQ):** не пришёл ACK за время T → шлём ещё раз.
- **Нумерация (sequence numbers):** для обнаружения дубликатов и сортировки.
- **Контрольные суммы:** обнаружение повреждённых пакетов.

Чем сильнее гарантии, тем больше задержка (waiting for ACK), buffer и сложность.

## Пример
- **Надёжный + connection-oriented:** TCP — для HTTP, SSH, SMTP, баз данных. Файл доходит весь и в порядке.
- **Ненадёжный + connectionless:** UDP без обвязки — для DNS-запросов, NTP. Простой запрос-ответ; если потерялся — повторим.
- **Надёжный + connectionless:** UDP с ретраями на уровне приложения (например, gRPC с retry policies). Реже встречается, но бывает.
- **Ненадёжный + connection-oriented:** ATM-сети, RTP без гарантий, исторические L2-протоколы. Соединение есть, но потери допустимы.

| | Connection-oriented | Connectionless |
|---|---|---|
| **Надёжный** | TCP, SCTP | reliable UDP-обвязки |
| **Ненадёжный** | ATM, RTP-сессия | UDP, IP |

## Связи
- **Базируется на:** [[Протокол vs служба]], [[Соединение vs без соединения]].
- **Используется в:** [[TCP]] (надёжный), [[UDP]] (ненадёжный), [[ARQ]] (механика надёжности).
- **Соседи по уровню:** надёжность можно строить на любом уровне — L2 (PPP с ACK), L4 (TCP), L7 (приложение с retry).
- **Противопоставляется:** «всегда нужна надёжность» — заблуждение для real-time трафика, где задержка критичнее.

## Подводные камни
- Надёжность — **end-to-end** свойство, по принципу [[Цели проектирования сетей|end-to-end argument]]. Делать её по чуть-чуть на каждом канале (L2 ACK), но не end-to-end, — недостаточно для приложения.
- TCP надёжен только до **получения** в ОС, а не до того, что приложение прочитало или сохранило. Ваше приложение должно само решать, что делать дальше.
- Надёжная служба может **намеренно** дропнуть пакет (load shedding) при перегрузке. См. [[Перегрузка сети]].
