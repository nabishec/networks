# Одноразовый блокнот (One-Time Pad, OTP)

## TL;DR
Единственный шифр с **доказуемой абсолютной стойкостью** (Shannon, 1949). Ключ — **истинно случайная** последовательность той же длины, что и сообщение. Зашифрование = XOR (или mod-26 для текста). Каждый ключ используется **один раз**. На практике редко применим — проблема **распространения ключа** делает его непрактичным для большинства систем.

## Какую проблему решает
Идея: если ключ K истинно случайный и длиной как M, то C = M ⊕ K — статистически **равновероятно** любым возможным M. Атакующий, зная только C, не может выделить M из всех возможных. **Информационно-теоретическая стойкость** — даже бесконечный compute не поможет.

## Как работает

**Алгоритм:**
1. Сгенерировать ключ K — истинно случайный, той же длины что сообщение.
2. C = M ⊕ K.
3. Передать C по open channel. Передать K по **secure channel** (другой канал, никогда не повторяется).
4. Получатель: M = C ⊕ K.

**Условия абсолютной стойкости:**
- K **истинно случайный** (не PRNG!).
- K используется **один раз** — никогда не повторно.
- K **никогда не передаётся** через канал, который атакующий мог подслушать.

## Пример
**Холодная война, дипломатическая связь:**
- Посольство получало большой блокнот случайных страниц.
- Каждое сообщение — XOR с next page.
- Использованная страница — уничтожить.

**Совершенно секретные коммуникации между лидерами стран** иногда используют до сих пор для critical channels.

**Пример атаки на повторное использование:**
- Если K используется дважды: C1 = M1⊕K, C2 = M2⊕K.
- C1⊕C2 = M1⊕M2 → атакующий имеет XOR двух plaintext'ов.
- Языковые статистики позволяют вытянуть оба сообщения.

## Связи
- **Базируется на:** теория информации Шеннона.
- **Используется в:** crypto-теория (нижний bound), отдельные military/diplomatic каналы.
- **Соседи по уровню:** **stream ciphers** (RC4, ChaCha20) — pseudo-random keystream вместо truly random — практичны, но **computationally** secure (а не **information-theoretically**).
- **Противопоставляется:** все practical ciphers — computationally secure (можно сломать с достаточным compute, но за разумное время невозможно).

## Подводные камни
- **Истинная случайность** — сложно. Hardware RNG (radiation, thermal noise) — разные качества.
- **Key distribution** — главная проблема: ключ длины сообщения должен где-то обмениваться. Если есть secure channel для key — почему не использовать его для message?
- **Не подходит для интернета** — нужен ОДИН раз key того же размера, что и трафик.
- **Reuse атак** — один из самых разрушительных историй кампрования (Venona, Soviet).
