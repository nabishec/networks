# Проектирование хостов для быстрых сетей

## TL;DR
**На 100+ Гбит/с CPU становится узким местом, не сеть.** Mogul/Metcalfe правила (1990s): **минимизировать число пакетов** (большие сегменты, TSO), **минимизировать число копирований** payload'а (zero-copy), **минимизировать context switches** (interrupt coalescing, batching). Современные техники: **DPDK** (kernel bypass), **XDP/AF_XDP** (eBPF на NIC), **io_uring** (async I/O), **RDMA** (zero CPU на data plane).

## Какую проблему решает
До 2000-х узким местом была **сеть** (1, 10, 100 Мбит/с). С 2010+ — **CPU**: 100 Гбит/с = 7.5 миллиардов 64-байтных-пакетов/с теоретически. На обычном Linux-стеке per-packet overhead ~1 μs → CPU обработает ~1 млн пакетов/с. Получается, **сеть быстрее CPU**.

Архитектурный рефакторинг хостовых стеков — тема последних 15 лет.

## Как работает

### Главные узкие места классического Linux TCP

| Узкое место | Стоимость per packet |
|---|---|
| **Interrupt** | ~1 μs (NIC → kernel) |
| **Copy NIC ↔ kernel buffer** | hundreds ns |
| **Context switch user ↔ kernel** | ~1 μs |
| **TCP/IP stack walking** | ~500 ns |
| **Copy kernel ↔ user buffer** | hundreds ns |

**Cumulative:** ~3-5 μs per packet → ~200-300k pps на core.

### Решения

**1. Reduce number of packets**
- **TSO** (TCP Segmentation Offload): NIC сама режет на сегменты.
- **GRO** (Generic Receive Offload): NIC собирает фрагменты до stack-uplink.
- **Jumbo frames** (9000 байт) — fewer packets для same data.

**2. Zero-copy**
- `sendfile()`, `splice()` — данные идут из page-cache в NIC без копии в user-space.
- **mmap** для shared buffers.
- **RDMA** — DMA directly from one host's memory to another.

**3. Kernel bypass**
- **DPDK** (Intel, 2010): user-space-driver для NIC, polling вместо interrupts. Достигает 80+ Mpps на core.
- **XDP / AF_XDP** (Linux): eBPF-программы на самом раннем receive-path; zero-copy в user-space.
- **io_uring** (Linux 5.1+): async I/O без syscall'а на каждый op.

**4. Hardware offload**
- **TLS offload** — encryption на NIC.
- **TCP offload engine (TOE)** — full stack on NIC (контроверсный, не mainline в Linux).
- **SmartNIC / DPU** (NVIDIA Bluefield, AWS Nitro): полный stack offload в FPGA/ASIC.

**5. NUMA-aware placement**
- IRQ binding к нужному CPU core.
- Memory allocation на тот же NUMA-node.
- Receive Side Scaling (RSS) — пакеты идут в разные cores по hash 5-tuple.

```mermaid
flowchart LR
  NIC --interrupt--> Driver
  Driver --copy--> KernelBuf[kernel sk_buff]
  KernelBuf --tcp/ip stack--> Sock[socket buffer]
  Sock --copy--> UserBuf[user buffer]
  UserBuf --> App
```

С kernel bypass:
```
NIC --DMA--> UserBufDirect --> App
```

## Пример
**HFT (High-Frequency Trading) firm:**
- Требование: latency <10 μs end-to-end.
- Linux TCP: ~50 μs.
- Решение: **DPDK** + custom user-space TCP (mTCP, F-Stack, Seastar).
- Полная обработка: ~5 μs.

**CDN (Cloudflare):**
- Использует **AF_XDP** на edge-серверах для DDoS mitigation: фильтрация на early packet path до TCP-stack.
- **eBPF** для programmable filters.

**Hyperscale DC (Google, Facebook):**
- **RDMA** для DC-internal: zero-copy, kernel bypass.
- **SmartNICs** (Microsoft Catapult) для compute offload.

## Связи
- **Базируется на:** [[TCP]] (что оптимизируется), [[TCP fast path и header prediction]] (родственная idea на kernel-level).
- **Используется в:** HFT, CDN edge, hyperscale DC, telecom 5G UPF.
- **Соседи по уровню:** **RDMA over Ethernet (RoCE)**, **InfiniBand**, **NVLink** — выходящие за TCP scope.
- **Противопоставляется:** «обычный POSIX socket для всего» — не масштабируется на 100+ Гбит/с.

## Подводные камни
- **Kernel bypass теряет** kernel-features (firewall, traffic shaping, observability) — нужно делать в user-space.
- **Vendor lock-in:** DPDK работает с конкретными NIC. SmartNIC API — proprietary.
- **Dev complexity:** lock-free programming, NUMA-awareness, cacheline-alignment — сложно.
- **Не для всех:** простые web-API не нуждаются — premature optimization. DPDK имеет смысл при ≥10 Гбит/с/core.
