# Сжатие заголовков (header compression)

## TL;DR
В мобильных и некоторых embedded-сценариях **заголовки** (IP+TCP+TLS+HTTP) могут быть в разы больше **полезной нагрузки** (например, VoIP-пакет: 40 байт payload + 60 байт заголовков). **Сжатие заголовков** использует тот факт, что соседние пакеты в одной сессии имеют **очень похожие** заголовки → передавать только различия. **ROHC** для real-time over wireless; **HPACK** для HTTP/2; **QPACK** для HTTP/3.

## Какую проблему решает
В беспроводных сетях полоса дорогая, голос/видео имеют маленький payload (10-40 байт), header-overhead — критичный фактор экономики. На мобильной сети ROHC может удвоить число одновременных VoIP-вызовов.

## Как работает

**Принцип «контекста»:**
- Передающая и принимающая стороны хранят **контекст** (state) сжатия — последний полный заголовок.
- Новый пакет → сравнить с контекстом → передать только изменения (delta).
- Коды для типичных полей: incrementing seq, постоянный IP, постоянный port.

**Уровни сжатия в ROHC (RFC 5795):**
- **U (Uncompressed):** initialization, full header.
- **R (Reliable):** ACK'и от приёмника подтверждают context — можно агрессивнее сжимать.
- **O (Optimistic):** assume context valid; recovery если рассогласование.

**Степень сжатия:**
- 60 байт IPv6+UDP+RTP → 1-3 байта в установившейся фазе.

**HPACK (HTTP/2, RFC 7541):**
- Static table: топ-61 most-common headers (`:method GET`, `content-type` и т.д.) — короткие индексы.
- Dynamic table: добавляются заголовки сессии (Cookie, Authorization).
- Huffman coding для строк.
- Уязвимость **CRIME**: атакующий может выявить секреты по размеру сжатого ответа → HPACK ограничивает что можно сжимать.

**QPACK (HTTP/3, RFC 9204):**
- Адаптация HPACK для QUIC: dynamic table обновляется отдельным stream'ом, чтобы не было HoL-blocking.

## Пример
- **VoIP в LTE:** ROHC активирован на radio bearer. Пакет 60 байт payload + 60 заголовков → 60 + 3 = 63 байта overhead'а. **2× больше клиентов** в той же радиополосе.
- **Web-browser:** HTTP/2 multiplexes 100 запросов на одно соединение. Каждый имеет 100+ заголовков, 80% повторяются (User-Agent, Accept). HPACK уменьшает overhead в 5-10 раз.

## Связи
- **Базируется на:** [[TCP]], [[IPv4]] (что сжимаем), хранение state в обоих концах.
- **Используется в:** **ROHC** (mobile carriers — LTE/5G U-plane), **HPACK** (HTTP/2), **QPACK** ([[QUIC]]/HTTP/3).
- **Соседи по уровню:** **deflate/gzip/Brotli** — сжатие body, не headers.
- **Противопоставляется:** «без сжатия» — простой, но дорогой на короткий payload.

## Подводные камни
- **Context loss** (после long idle или packet drop) → не можем декомпрессировать → recovery (full headers, or session reset).
- **CRIME-class atacks**: сжатие не должно смешивать секреты с публичными данными.
- ROHC требует **stateful** middleboxes — мобильное ядро (PGW) держит per-flow context. На WAN-краю сложнее.
