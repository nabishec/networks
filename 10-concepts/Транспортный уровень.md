# Транспортный уровень (Transport Layer, L4)

## TL;DR
Делает сеть **полезной для приложений**: добавляет к ненадёжному IP **порты** (множество приложений на одном хосте), **надёжность** (TCP), **управление потоком и перегрузкой**. Главные представители: **TCP** (надёжный, упорядоченный, byte stream) и **UDP** (без гарантий, для real-time/short queries). End-to-end: участвуют **только хосты**, не маршрутизаторы.

## Какую проблему решает
IP доставляет пакеты от хоста к хосту, без гарантий: пакет может потеряться, прийти не по порядку, продублироваться. Приложение должно либо смириться с этим (UDP+своя логика), либо получить надёжность поверх (TCP). L4 — это слой, на котором эта надстройка реализуется.

## Как работает

**Главное на L4 (по Tanenbaum'у §6.1–6.2):**

1. **Адресация:** [[Порт]] — 16 бит ID приложения. Полный 5-tuple `(src IP, src port, dst IP, dst port, protocol)` идентифицирует **поток**.
2. **Установление и разрыв соединения** (для CO-протоколов): [[Three-way handshake]], FIN/RST.
3. **Контроль ошибок:** sequence numbers + retransmission ([[ARQ]]).
4. **Управление потоком:** flow control — получатель не утонул в скорости отправителя.
5. **Контроль перегрузки:** congestion control — сеть не утонула.
6. **Мультиплексирование:** один сетевой стек обслуживает много приложений и соединений.

**API:** [[Сокеты Беркли]] — `socket()`, `bind()`, `listen()`, `accept()`, `connect()`, `send()`, `recv()`, `close()`.

## Пример
**Открыть `https://wikipedia.org`:**
- Браузер вызывает `socket(AF_INET, SOCK_STREAM)` → TCP-сокет.
- `connect()` к 208.80.154.224:443 → ОС инициирует [[Three-way handshake]].
- Параллельно может быть `socket(AF_INET, SOCK_DGRAM)` для DNS-запроса по UDP.
- Один процесс, один хост — много сокетов разных типов и направлений.

## Связи
- **Базируется на:** [[Сетевой уровень]] (несётся в IP-пакетах).
- **Используется в:** [[TCP]], [[UDP]], [[QUIC]] — основные протоколы; [[Сокеты Беркли]] — API.
- **Соседи по уровню:** [[Сетевой уровень]] (ниже), [[Web — архитектура]] и др. прикладные (выше).
- **Противопоставляется:** L3 — hop-by-hop через сеть; L4 — end-to-end на хостах.

## Подводные камни
- **End-to-end argument:** надёжность лучше всего реализуется на L4 у хоста, не на L3 у маршрутизатора.
- На L4 живут и **flow control** (rwnd), и **congestion control** (cwnd) — это **разные** механизмы с разными целями.
- Маршрутизаторы, **в идеале**, не вскрывают L4-заголовок. Реальность — NAT, firewall, L4-loadbalancer регулярно вскрывают.
