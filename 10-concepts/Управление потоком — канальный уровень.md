# Управление потоком — канальный уровень

## TL;DR
Защита **получателя** от слишком быстрого отправителя на одном канале. Два подхода: **feedback-based** — получатель сообщает «жду столько-то ещё» (rwnd, sliding window); **rate-based** — отправитель ограничен заранее согласованной скоростью. На L2 в современных сетях (Ethernet) feedback преобладает в специальных случаях (PAUSE-фреймы для DC). На L4 — TCP rwnd.

## Какую проблему решает
Получатель имеет **buffer** конечного размера. Если отправитель шлёт быстрее, чем приёмник обрабатывает → buffer overflow → потери. Управление потоком — механизм согласования темпа.

**Не путать с [[Перегрузка сети|congestion control]]:** flow control защищает **получателя**; congestion — **сеть**.

## Как работает

### Feedback-based (закрытая петля)
- Получатель **информирует** отправителя о доступной ёмкости.
- Sliding window: rwnd в каждом ACK.
- IEEE 802.3x **PAUSE frame** в Ethernet: получатель шлёт PAUSE → отправитель пауза N квантов.
- Lossless Ethernet (DCBX, PFC): в дата-центрах, где нужна low-latency без drop'ов.

### Rate-based (открытая петля)
- Отправитель ограничен заранее (token bucket, leaky bucket).
- Не требует обратной связи.
- Подходит для **broadcast** или сетей без feedback (спутник в одну сторону).
- Проблема: если получатель медленнее, чем rate → потери. Нужна **conservative** оценка.

### Сравнение

| | Feedback-based | Rate-based |
|---|---|---|
| Реактивность | да (адаптируется) | нет (статичен) |
| Требует обратный канал | да | нет |
| Простота | сложнее | проще |
| Применение | TCP, sliding-window | broadcast, RT-streaming |

## Пример
**Lossless Ethernet в DC** (RDMA-traffic):
- На loaded port switch шлёт PAUSE-фрейм отправителю.
- Отправитель приостанавливается (микросекунды).
- Когда buffer освобождается — port не PAUSE'и.
- В сочетании с **PFC** (Priority Flow Control) — per-priority paused.
- Используется RoCE (RDMA over Converged Ethernet).

**Старые HDLC-сети** на serial-линках использовали windowed feedback.

## Подводные камни
- **PAUSE-фреймы могут вызвать head-of-line blocking** на switch'е — если один port paused, трафик к нему блокирует и других.
- **PFC + deadlock** — известная проблема в RoCE-сетях; circular paused может застрять.
- В general internet (вне DC) PAUSE не используется — drop tail + TCP CC решают.
