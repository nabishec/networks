# Цели проектирования сетей

## TL;DR
Семь повторяющихся целей, в напряжении между которыми живут все сетевые архитектуры: **надёжность**, **масштабируемость**, **управление ресурсами**, **безопасность**, **эволюция**, **совместимость**, **интероперабельность**. Любое решение в сети — это компромисс между ними.

## Какую проблему решает
Сети растут быстрее, чем меняются их фундаментальные протоколы. IP создавался для сотен узлов — сегодня их миллиарды. Чтобы архитектура переживала такие сдвиги, проектировщики держат в голове набор **целей**, по которым любое решение проверяется. Без этого получается «работает в этой версии — ломается в следующей».

## Как работает

| Цель | Что значит | Пример напряжения |
|---|---|---|
| **Надёжность** | не теряет данные, не теряет связь, корректен при ошибках | надёжность ↔ задержка (ACK, ретрансмиссии) |
| **Масштабируемость** | хорошо работает при росте узлов в 10×, 100×, 1000× | flat-адресация удобна, но не масштабируется → CIDR |
| **Управление ресурсами** | пропускная способность делится справедливо, перегрузка не убивает сеть | TCP-fairness, [[Перегрузка сети]] |
| **Безопасность** | конфиденциальность, целостность, доступность ([[Триада CIA]]) | шифрование ↔ задержка/CPU |
| **Эволюция** | можно вводить новое без поломки старого | IPv4 → IPv6 десятилетиями |
| **Совместимость** | старое железо и софт продолжают работать | NAT как костыль для IPv4-дефицита |
| **Интероперабельность** | разные вендоры/реализации работают вместе | RFC, IETF-процесс |

Несколько повторяющихся **архитектурных принципов**, помогающих достигать целей:
- **End-to-end argument** — функции, нужные приложению (надёжность, шифрование), лучше реализовывать на конечных точках, а не в сети.
- **Layering** — см. [[Уровневая архитектура]].
- **Идемпотентность** — операция при повторении даёт тот же результат → проще ретраи.
- **Soft state** — состояние, которое истекает по таймеру → не нужно явно чистить при сбоях.
- **Fate sharing** — состояние и компонент должны жить/умирать вместе.

## Пример
- **End-to-end:** надёжность даёт TCP на хостах, не маршрутизаторы. Маршрутизатор может потерять пакет — TCP перешлёт.
- **Soft state:** ARP-кэш живёт минуты и обновляется. Если запись «сбилась» — после таймаута запросится заново. Не нужно вручную чистить.
- **Эволюция:** IPv4 не уходит, потому что миллиарды устройств; параллельно IPv6 растёт, dual-stack — норма десятилетие.

## Связи
- **Базируется на:** [[Компьютерная сеть]] — это её свойства, к которым стремимся.
- **Используется в:** обоснование почти любой архитектурной заметки книги.
- **Соседи по уровню:** [[Уровневая архитектура]] (механизм), [[Триада CIA]] (детализация безопасности).
- **Противопоставляется:** наивный взгляд «сделать максимально быстро» — в крупной сети так не работает.

## Подводные камни
- Цели **в конфликте**. Шифрование улучшает безопасность, но ухудшает производительность; flat-адресация удобна, но плохо масштабируется. «Лучшее» — компромисс под задачу.
- End-to-end argument **не догма**. NAT, MPLS, CDN — все нарушают его и при этом полезны. Принципы — не запреты, а ориентиры.
