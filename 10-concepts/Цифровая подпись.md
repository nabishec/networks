# Цифровая подпись (digital signature)

## TL;DR
**Asymmetric**-механизм: подписант **подписывает** хэш сообщения **своим private key** → любой с **public key** может **проверить** подпись. Свойства: **authenticity** (от того, кто подписал), **integrity** (не изменено), **non-repudiation** (подписант не может отказаться). Базис PKI, code signing, blockchain. Алгоритмы: RSA-PSS, ECDSA, Ed25519, EdDSA.

## Какую проблему решает
HMAC даёт integrity + authentication, но **обе стороны** знают ключ → нет non-repudiation. Цифровая подпись: только подписант имеет private key, никто другой не может подделать → бесспорное доказательство «именно он подписал».

## Как работает

**Подпись:**
1. Подписант: `hash = H(message)`.
2. `signature = sign(private_key, hash)`.
3. Передать сообщение + signature.

**Проверка:**
1. Получатель: `hash = H(message)`.
2. `verify(public_key, hash, signature)` → boolean.
3. True → подпись валидна, сообщение от владельца private_key и не изменено.

**Алгоритмы:**

| Алгоритм | Тип | Размер ключа | Скорость |
|---|---|---|---|
| **RSA-PSS** | RSA-based | 2048-4096 бит | средняя |
| **DSA** | Discrete log | 2048+ | медленная подпись |
| **ECDSA** | Elliptic curve | 256-521 бит | быстрая |
| **Ed25519** | Edwards curve | 256 бит | очень быстрая, modern |
| **ML-DSA (Dilithium)** | Post-quantum | большие | будущее |

**Hash-then-sign:** сообщение хешируется, потом подписывается hash. Иначе слишком долго на больших файлах + ограничения block-cipher size.

## Пример
**HTTPS-сертификат:**
- Сертификат example.com содержит: subject=example.com, public_key, expiry, и т.д.
- CA подписывает (e.g., Let's Encrypt R3) **своим** private key.
- Браузер: проверяет с public_key R3 (известным из root store).

**Code signing (Apple, Microsoft):**
- Разработчик подписывает приложение своим private.
- ОС проверяет с public, известным от Apple/Microsoft.
- Не подписано / подпись invalid → warning.

**Blockchain (Bitcoin):**
- Каждая транзакция подписана private key владельца coins.
- Network проверяет подписи.

## Связи
- **Базируется на:** [[Симметричная vs асимметричная криптография]], [[Хеш-функции]], [[RSA]] (или ECDSA, Ed25519).
- **Используется в:** [[X.509 сертификаты]], TLS-handshake, JWT-RS, code signing, blockchain.
- **Соседи по уровню:** [[HMAC]] — symmetric MAC, без non-repudiation.
- **Противопоставляется:** HMAC — обе стороны знают ключ; sign — только подписант имеет private.

## Подводные камни
- **Никогда не подписывай arbitrary message** — security модели часто требуют structured messages (e.g., DSS — domain separation tags).
- **Signature ≠ encryption.** RSA-encrypt и RSA-sign — разные операции (PKCS#1 v1.5 vs PSS).
- **Quantum threat:** RSA, ECDSA уязвимы. **ML-DSA, SLH-DSA** — post-quantum signatures, NIST-стандартизованы 2024.
- **Time-of-signing matters:** signature, valid сейчас, может стать невалидной (revoked cert). **OCSP, CRL** — для проверки revocation.
