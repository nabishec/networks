# BGP — Border Gateway Protocol (RFC 4271)

## TL;DR
**Главный inter-AS протокол** интернета: соединяет автономные системы (~75 000 AS) в единую глобальную сеть. **Path-vector** (расширение Distance Vector с явным AS-path для устранения петель). Принимает решения **на основе политики**, не shortest path: оператор выбирает маршруты по бизнес-причинам (транзит/пиринг/customer). Без BGP интернет не интернет.

## Какую проблему решает
[[Distance Vector Routing|RIP]] и [[Link State Routing|OSPF]] хороши **внутри** одной организации, где все доверяют друг другу и используют общие метрики. Между AS:
- Доверия нет — каждый оператор свой.
- Метрики разные.
- Решения **политические** (с кем пиринг, кому транзит, что фильтровать).
- Нужна агрегация на масштабе миллиарда хостов.

BGP даёт: protocol для обмена reachability + механизмы политики.

## Как работает

**Path-vector:** каждое объявление содержит **AS_PATH** — список AS, через которые маршрут прошёл. Например, `203.0.113.0/24 — AS_PATH: 65000 65001 65002`. Помогает:
- **Устранять петли:** «если я уже в AS_PATH — не принимать обратно».
- **Выбирать кратчайший** при равных политиках.

**eBGP vs iBGP:**
- **eBGP** — между AS. По умолчанию TTL=1 для прямых соединений (RFC 4271 / Cisco best practice; `ebgp-multihop` отключает).
- **iBGP** — внутри AS, для распространения eBGP-маршрутов между PE-роутерами.

**Атрибуты маршрута** (главные):
- **AS_PATH** — последовательность AS.
- **NEXT_HOP** — IP next-hop.
- **LOCAL_PREF** — внутри AS, выше = лучше.
- **MED** (Multi-Exit Discriminator) — подсказка соседнему AS.
- **COMMUNITY** — теги для политики (RFC 1997). Например, "не объявлять за пределами Европы".

**Алгоритм выбора лучшего пути** (упрощённо, в порядке):
1. Highest weight (Cisco-specific, локально).
2. Highest LOCAL_PREF.
3. Locally originated.
4. **Shortest AS_PATH.**
5. Lowest origin code.
6. Lowest MED.
7. eBGP > iBGP.
8. Lowest IGP cost to next-hop.
9. Tie-breakers (lowest router ID и т.д.).

**Декларативная политика:**
- **Customer:** платит за транзит → объявлять customer-маршруты всем.
- **Peer:** взаимный (settlement-free) → объявлять peer-маршруты только customer'ам.
- **Provider:** мы платим → не объявлять чужому upstream'у.

## Пример
**Российский провайдер MTS:**
- Имеет AS 8359.
- Пиринг с другими российскими (Ростелеком, Билайн) и зарубежными.
- Получает full table (~1M маршрутов) от своих transit-провайдеров.
- Объявляет свои префиксы (адреса абонентов, серверов) всему миру через transit.
- Реализует политику фильтрации: не принимать chinese-AS если CN-сайт offline; не объявлять peer-маршруты transit'у; и т.д.

## Связи
- **Базируется на:** [[Distance Vector Routing]] (path-vector — расширение DV), [[Иерархическая маршрутизация]] (AS как слой иерархии), [[Транзитная сеть]] (экономика).
- **Используется в:** все межоператорские стыки, [[Anycast]] (через анонс одного префикса из многих мест), [[MPLS]]-VPN (через VPNv4/VPNv6 BGP).
- **Соседи по уровню:** [[OSPF]], IS-IS — IGP внутри AS; BGP — EGP между.
- **Противопоставляется:** IGP — внутри одной организации, общая метрика; BGP — между, политика.

## Подводные камни
- **BGP hijacking** — один из главных рисков интернета. Кто угодно может анонсировать чужой префикс. Защита — **RPKI** (Resource Public Key Infrastructure), валидация Origin AS.
- **Конвергенция медленная** в сравнении с OSPF — десятки секунд при крупных изменениях. Компромисс масштабируемости.
- **Slow convergence + traffic engineering errors** — частая причина региональных инцидентов в интернете.
- BGP — **не shortest path**. Маршрут с короче AS_PATH часто проигрывает «политически предпочтительному». Это норма.
- **Internet routing table** растёт ~10% в год. Маршрутизаторы провайдеров — это серьёзная инвестиция в железо.
