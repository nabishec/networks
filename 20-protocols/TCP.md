# TCP (Transmission Control Protocol, RFC 9293)

## TL;DR
TCP делает из ненадёжной сети **надёжный канал** — как телефонный звонок: позвонил («алло»-«алло»-«говори»), поговорили без потерь и в правильном порядке, попрощались. Под капотом: нумерует кусочки данных, переспрашивает потерянные, тормозит при перегрузке. Главный протокол интернета — основа HTTP, SSH, SMTP, баз данных. Альтернатива — **простой и быстрый, без гарантий** [[UDP]] (как почтовая открытка).

## Какую проблему решает
IP — best-effort: пакет может потеряться, прийти не по порядку, дублироваться. Большинство приложений (HTTP, SSH, БД) не хотят это решать сами. TCP даёт **надёжный поток** — приложение пишет байты, читает байты, всё «как файл».

## Как работает

**Главные компоненты:**

1. **Установление:** [[Three-way handshake]] (SYN, SYN-ACK, ACK).
2. **Структура:** [[TCP — заголовок]] (порты, seq, ack, флаги, window).
3. **Передача:** [[TCP — раздвижное окно]] (rwnd для flow control, cwnd для congestion).
4. **Надёжность:** sequence numbers + cumulative ACK (опционально + SACK).
5. **Контроль перегрузки:** [[TCP — slow start]] → [[TCP — congestion avoidance]] (AIMD), при потерях — fast retransmit / fast recovery.
6. **Закрытие:** [[Разрыв соединения]] (FIN, ACK; полузакрытие).
7. **Состояния:** [[TCP — состояния]] (CLOSED, LISTEN, ESTABLISHED, TIME_WAIT, …).

**Основные гарантии для приложения:**
- **Reliability:** все байты доставлены или соединение разорвано.
- **In-order:** байты приходят в порядке отправки.
- **No duplicates:** дубли отбрасываются.
- **Flow control:** получатель не утонет.
- **Congestion control:** сеть не утонет.

**Не гарантирует:**
- **Boundary preservation:** TCP — поток, не сообщения. Если отправил два `send(N байт)`, получатель может получить одним `recv()` или несколькими — границы стираются.
- **Latency:** доставит — но не за столько-то мс.

## Пример
**HTTP-запрос:**
1. Клиент: `connect()` → 3-way handshake.
2. `send("GET / HTTP/1.1\r\n\r\n")` → TCP-сегмент.
3. Сервер `recv()` → читает байты → парсит HTTP.
4. `send(...response...)` → возможно много сегментов.
5. Клиент `recv()` → собирает весь ответ.
6. `close()` → 4-way teardown.

**Современная эра:** часто HTTP/2 на одном TCP-соединении мультиплексирует много запросов; HTTP/3 ушёл на QUIC поверх UDP — TCP head-of-line blocking стал болью.

## Связи
- **Базируется на:** [[IPv4]] / [[IPv6]] (несётся в IP), [[ARQ]] (надёжность), [[Скользящее окно]] (механика).
- **Используется в:** HTTP/HTTPS (до 2.0), SSH, SMTP/IMAP, FTP, базы данных, gRPC.
- **Соседи по уровню:** [[UDP]] — без гарантий, [[QUIC]] — современная альтернатива поверх UDP.
- **Противопоставляется:** UDP — short messages без гарантий; QUIC — TCP+TLS+мультиплексирование без HoL.

## Подводные камни
- **Head-of-Line blocking:** один потерянный сегмент задерживает все следующие, даже если они в буфере. На HTTP/2 это критично → HTTP/3 ушёл на QUIC.
- **Slow start** — медленный старт; для коротких HTTP-запросов часто не успевает разогнаться. Initial cwnd увеличивали (RFC 6928).
- **TIME_WAIT** на закрытии забивает порты на сервере под нагрузкой.
- **TCP надёжен только до kernel получателя**; приложение должно само решать persistence.
- **MSS, window scale, SACK** — обязательные опции для современных high-BDP сетей.
