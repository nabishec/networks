# UDP — User Datagram Protocol (RFC 768)

## TL;DR
**Минимальный** транспортный протокол: 8-байтный заголовок (src port, dst port, length, checksum), всё. Без соединения, без подтверждений, без reordering, без потоков. **Best-effort**: пакет может потеряться, прийти не по порядку, продублироваться. Зачем такой? Скорость, простота, точечные запросы (DNS), real-time (RTP), broadcast/multicast, основа для **QUIC**.

## Какую проблему решает
TCP — мощный, но дорогой: handshake (1 RTT), state на хостах, retransmissions блокируют поток. Не для всех задач:
- DNS-запрос 50 байт → handshake съедает больше, чем сам запрос.
- VoIP/видео → ретрансмиссия устаревшего пакета вреднее потери.
- Multicast → TCP не поддерживает.

UDP даёт «голый IP с портами» — приложение реализует своё.

## Как работает

**Заголовок UDP (8 байт):**
```
+--------+--------+--------+--------+
| Source Port    | Dest Port       |
+--------+--------+--------+--------+
| Length          | Checksum       |
+--------+--------+--------+--------+
|         payload (0-65527)         |
```

- **Source port** — может быть 0 (нет ответа).
- **Dest port** — обязательно.
- **Length** — UDP-заголовок + payload (8..65535).
- **Checksum** — сумма pseudo-header (src/dst IP) + UDP-заголовок + payload. В IPv4 опциональна (=0 → не проверяется); в IPv6 обязательна.

**Без:**
- Connection setup.
- Acknowledgements.
- Sequence numbers.
- Retransmission.
- Flow control.
- Congestion control.

Если приложению нужно — реализует **сверху**.

## Пример
- **DNS-запрос:** клиент → 8.8.8.8:53 один UDP-пакет. Ответ — один UDP-пакет. Если потерялся — клиент сам перешлёт.
- **NTP, DHCP, SNMP** — короткие запрос-ответ.
- **VoIP/RTP:** real-time медиа, потери допустимы, retransmission бесполезна (поздно).
- **Multiplayer games:** каждый кадр — UDP, важна свежесть, не последовательность.
- **QUIC** ([[QUIC]]): надёжный поток поверх UDP — TCP-подобный сервис, реализованный в user-space.

## Связи
- **Базируется на:** [[Транспортный уровень]], [[Internet checksum]] (контроль ошибок).
- **Используется в:** [[DNS]], [[RTP]], [[QUIC]], DHCP, NTP, SNMP, multicast-стримы.
- **Соседи по уровню:** [[TCP]] — антагонист со всеми гарантиями.
- **Противопоставляется:** TCP — сравнивай характеристики сразу.

## Подводные камни
- UDP **не имеет** congestion control → может **затопить** сеть. Современные UDP-приложения должны реализовать свой (DCCP, QUIC, RTCP-feedback).
- **Размер**: фрагментация UDP-пакета на L3 при больших пакетах хрупка. Современный совет: <1500-overhead, чтобы помещаться в один Ethernet-фрейм.
- **NAT и UDP**: state в NAT истекает быстрее (нет close-сигнала). Keepalive нужен для долгих UDP-сессий.
