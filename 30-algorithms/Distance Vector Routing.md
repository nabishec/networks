# Distance Vector Routing

## TL;DR
Distributed-алгоритм маршрутизации (Bellman-Ford): каждый маршрутизатор хранит **вектор** «до X — N хопов через сосед Y», обменивается этими векторами с **прямыми соседями** периодически. Получив чужой вектор, обновляет свой. Просто, но **медленно сходится** при изменениях, страдает от [[Count-to-infinity problem]]. Базис RIP и BGP (path-vector — расширение DV).

## Какую проблему решает
Хранить полную топологию сети (как в Link State) — дорого. DV даёт более скромное требование: каждый знает только своих **прямых соседей**. Распространение информации идёт постепенно через обмен векторами.

## Как работает

**Структура у маршрутизатора R:**
- Таблица: для каждого destination D — `(distance, next-hop)`.
- Знает «расстояния до соседей» (1 хоп или метрика канала).

**Обновление:**
1. Периодически (каждые ~30 с в RIP) каждый маршрутизатор шлёт свой вектор соседям.
2. Получив вектор от соседа N: для каждого D в векторе вычисляет `cost(R→N) + N.dist(D)`. Если меньше текущего R.dist(D) — обновляет, ставит next-hop = N.

**Bellman-Ford принцип:** dist(D) = min(over neighbors N) [cost(R→N) + N.dist(D)].

```
        |  A  |  B  |  C  |  D
A       |  0  |  2  |  5  |  3
B       |  2  |  0  |  3  |  1
C       |  5  |  3  |  0  |  2
D       |  3  |  1  |  2  |  0
```

(каждая строка — что знает узел; через обмен таблицы постепенно совпадают со shortest paths)

## Пример
**RIP** (Routing Information Protocol, RFC 1058) — классический DV:
- Метрика — число хопов, max 15 (16 = недостижимо, защита от петель).
- Updates каждые 30 с broadcast'ом всем соседям.
- Конвергенция при изменениях — десятки секунд.

**EIGRP** (Cisco) — улучшенный DV с loop-prevention (DUAL).

## Связи
- **Базируется на:** [[Принцип оптимальности маршрутизации]] (Bellman принцип); граф с весами.
- **Используется в:** RIP (старый IGP), EIGRP (Cisco), [[BGP]] (path-vector — расширение DV с явным AS-path).
- **Соседи по уровню:** [[Link State Routing]] — другая парадигма (полная топология vs только соседи).
- **Противопоставляется:** Link State — DV проще на маршрутизаторе, медленнее при изменениях; LS наоборот.

## Подводные камни
- **Count-to-infinity** ([[Count-to-infinity problem]]): когда канал отказывает, информация о недостижимости распространяется медленно через ложные обходные пути.
- Защиты: split horizon (не сообщать соседу маршрут, который через него идёт), poison reverse (сообщать с метрикой ∞), hold-down timers.
- Periodic full-table updates → много паразитного трафика. Современный RIPng уменьшает это, но всё равно DV проигрывает LS на крупных сетях.
