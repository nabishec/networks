# Internet checksum

## TL;DR
Простая 16-битная контрольная сумма для **детектирования** ошибок в IP/TCP/UDP. Складывает все 16-битные слова сообщения с **арифметикой обратного кода** (one's complement), затем берёт побитовое дополнение. На приёме сумма всех слов плюс checksum должна давать «все единицы». Слабее [[CRC — циклический избыточный код|CRC]], но достаточно дёшев, чтобы ОС могла считать его в софте.

## Какую проблему решает
Канальный уровень обычно ставит сильную CRC (например, CRC-32 в Ethernet). Зачем тогда ещё одна сумма в IP/TCP/UDP? **End-to-end argument:** L2-CRC защищает от ошибок в линии, но не от ошибок в маршрутизаторе (повреждение в памяти, баг в драйвере). Сетевой и транспортный уровни нужна **независимая** контрольная сумма, которая считается end-to-end.

Internet checksum выбран как **компромисс**: слабее CRC (пропускает некоторые паттерны), но в десятки раз быстрее в программной реализации — это критично для маршрутизаторов 1980-х и для каждого пакета сегодня.

## Как работает

**Алгоритм:**
1. Разбить сообщение на 16-битные слова.
2. Сложить их с **переносом обратно в младший бит** (one's complement addition).
3. Взять **побитовое дополнение** (`~`) результата — это и есть checksum.
4. На приёме: добавить полученный checksum к сумме слов; результат должен быть `0xFFFF` (все единицы) — ошибок нет.

```python
def checksum(data: bytes) -> int:
    s = 0
    for i in range(0, len(data), 2):
        word = (data[i] << 8) | data[i+1]
        s += word
        if s > 0xFFFF:
            s = (s & 0xFFFF) + 1   # carry-back
    return ~s & 0xFFFF
```

**Особенность one's complement:** существует два представления нуля (`+0` = `0x0000` и `−0` = `0xFFFF`). Поле checksum 0 в TCP/UDP означает «не использовался». Если реальная сумма получилась 0, её записывают как `0xFFFF`.

## Пример
**IP-заголовок:** 20 байт = 10 16-битных слов. Поле checksum — **только** для заголовка, не для payload.

```
Заголовок IP с полем checksum:
[Version | IHL | DSCP | ECN | Total Length | Identification | Flags | Fragment Offset | TTL | Protocol | HEADER CHECKSUM | Src IP | Dst IP]
```

**TCP/UDP checksum:** покрывает весь сегмент **+ pseudo-header** (включает Src/Dst IP и Protocol). Это защищает от misdelivery — если IP подменил адрес, TCP-checksum не сойдётся.

**Различия с CRC:**

| Свойство | Internet checksum | CRC-32 |
|---|---|---|
| Размер | 16 бит | 32 бита |
| Сложность | очень дешёвый, software | требует таблицы или sht/xor |
| Detection | пропускает некоторые ошибки (например, swap of equal-sized chunks) | **гарантирует** ловлю любых ≤32-битных burst |
| Где | IP, TCP, UDP, ICMP | Ethernet, Wi-Fi, PPP |

## Связи
- **Базируется на:** одна из простейших контрольных сумм; арифметика обратного кода.
- **Используется в:** [[IPv4]], [[TCP]], [[UDP]], ICMP.
- **Соседи по уровню:** [[CRC — циклический избыточный код]] — сильнее, но на L2.
- **Противопоставляется:** криптографические хэши (SHA-2) — против злого умысла; Internet checksum защищает от случайных ошибок.

## Подводные камни
- IPv6 **отказался** от заголовочного checksum в IP — TCP/UDP всё равно содержат свою сумму, IP-checksum только дублировал.
- Современный TCP-checksum часто **offloaded** в NIC (TSO/TCP segmentation offload). Драйвер ставит «псевдо-checksum», NIC доделывает. Можно увидеть `0x0000` в Wireshark из-за этого.
- Internet checksum не ловит **перестановку** двух 16-битных слов одинакового значения. Это известная слабость; на практике редкая.
