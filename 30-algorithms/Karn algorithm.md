# Karn's algorithm (Phil Karn, 1987)

## TL;DR
Решает проблему **измерения RTT при ретрансмиссиях** в TCP. Если сегмент послан, потом переслан, и пришёл ACK — **неизвестно, к какому из двух относится**. Karn: **не используй RTT-выборки от ретрансмитированных сегментов** (просто игнорируй); и при таймауте **удваивай RTO** до получения честной выборки. Простое, изящное, во всех TCP-стэках.

## Какую проблему решает
TCP оценивает RTT по разнице время-ack-time-send. Это нужно для RTO (Retransmission TimeOut). Если сегмент шлётся два раза, и приходит ACK:
- ACK на первый? Тогда измеренный RTT = реальный RTT.
- ACK на второй? RTT = только время после ретранса (мало) → недооценка.

Без защиты — RTO «ползёт вниз», ложные ретрансы, спираль.

## Как работает

**Правило 1 (sample skipping):**
- Если сегмент с seq=X отослан и потом переслан → не используй RTT-выборку от ACK на X.
- Берём только «честные» сэмплы — от сегментов, отосланных один раз.

**Правило 2 (timer backoff):**
- При таймауте: новое RTO = 2 * старого RTO.
- Не сбрасывать на основе ACK во время backoff.
- При следующей честной выборке RTO пересчитывается обычно.

**Зачем оба:**
- Sample skipping без backoff — на устойчивой потере пропадают все выборки, RTO застрянет.
- Backoff обеспечивает экспоненциальный рост RTO при упорных потерях (как в CSMA/CD).

## Пример
**Сценарий:**
- t=0: send seq=1000.
- t=1.0: timeout (RTO=1.0), retransmit seq=1000.
- t=1.2: ACK на seq=1000.

**Без Karn:** RTT=0.2, странно мало → RTO будет уменьшен.
**С Karn:** sample игнорируется. RTO = 2.0 (удвоение).
- t=1.5: send seq=1500 (новый).
- t=2.0: ACK на 1500. RTT=0.5 — честный → RTO пересчитывается.

## Связи
- **Базируется на:** [[TCP]], общая теория ARQ.
- **Используется в:** все TCP-стэки; концепт распространим на любой [[ARQ]]-протокол.
- **Соседи по уровню:** **Jacobson's algorithm** для расчёта RTO (RFC 6298): srtt + 4*rttvar; Karn применяется поверх.
- **Противопоставляется:** наивная оценка RTT без учёта ретрансмиссий — нестабильна.

## Подводные камни
- При **постоянных** потерях RTO растёт экспоненциально → bouts of slow connectivity. Это норма, но больно.
- **Timestamps option** (RFC 1323) частично решает: каждый сегмент имеет timestamp → можно различать по timestamp ACK'а.
- В **QUIC** Karn-логика тоже есть, но проще — каждый пакет имеет уникальный packet number, нет ambiguity при ретрансмиссиях.
