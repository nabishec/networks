# Диффи-Хеллман (1976)

## TL;DR
Первый алгоритм для **установления общего секретного ключа через open channel**. Алиса и Боб обмениваются «public» значениями; через простую mathematics получают **общий секрет**, который никто не может вычислить из перехваченного. Не для шифрования сам по себе — только для **key exchange**. Современные варианты: **ECDH** (на эллиптических кривых), **DHE/ECDHE** (ephemeral для perfect forward secrecy).

## Какую проблему решает
Симметричная криптография требует общий секрет. **Как** его передать через подслушиваемый канал? До 1976 г. — никак (или secure channel — но тогда зачем шифровать?). Diffie-Hellman: **математически** установить секрет в open channel.

## Как работает

**Базовый DH (multiplicative group mod p):**

1. Public params: prime **p** и generator **g**.
2. Alice выбирает random `a`, считает `A = g^a mod p`. Шлёт A Bob.
3. Bob выбирает random `b`, считает `B = g^b mod p`. Шлёт B Alice.
4. Alice: `s = B^a mod p = g^{ab} mod p`.
5. Bob: `s = A^b mod p = g^{ab} mod p`.
6. Общий секрет — `s`.

Атакующий видит p, g, A, B → ему нужно решить **discrete logarithm** problem (найти a по A=g^a) — практически невозможно для p ~ 2048 бит.

**ECDH** (Elliptic Curve DH):
- То же, но в группе точек elliptic curve.
- Меньшие ключи (256 бит ≈ 3072-бит DH).

**DHE / ECDHE (Ephemeral):**
- Каждое соединение — **новые** случайные a, b.
- Если private key сервера украден позже, прошлые сессии **не расшифровываются** (нет state-key для decrypt).
- **Perfect Forward Secrecy** (PFS).
- Mandatory в TLS 1.3.

## Пример
**TLS 1.3 ECDHE-key-exchange:**
- Server и client посылают свои ECDHE-public values в Hellos.
- Каждый считает shared secret.
- Из shared secret деривируются session keys (HKDF).
- Если сервер's RSA private key позже compromised — прошлый трафик **не уязвим** (PFS).

**Diffie-Hellman MITM**:
- Без аутентификации DH уязвим к MITM (атакующий говорит обоим «я partner»).
- В TLS DH-параметры **подписаны** server'ом (RSA или ECDSA) → MITM не подделает signature.

## Связи
- **Базируется на:** discrete logarithm (DH), elliptic curve discrete log (ECDH).
- **Используется в:** [[TLS — рукопожатие]] (key exchange), [[IPsec]] (IKE), SSH, VPN.
- **Соседи по уровню:** [[RSA]] — другой подход (по факторизации).
- **Противопоставляется:** **PSK** (Pre-Shared Key) — secret уже общий из out-of-band.

## Подводные камни
- **Plain DH без аутентификации** — MITM. В реальности всегда **authenticated DH** (через signed parameters).
- **Параметры группы** — важны. Слабые primes → discrete log дешевле. Logjam-атака (2015) на small DH-параметры.
- **ECDH curve choice:** P-256 (NIST), Curve25519 (Bernstein) — современные стандарты. Backdoored curves (Dual_EC_DRBG) — печальная история NSA.
- **Quantum threat:** Shor ломает discrete log → DH/ECDH уязвимы. **Post-quantum KEM** (ML-KEM/Kyber) — на подходе.
